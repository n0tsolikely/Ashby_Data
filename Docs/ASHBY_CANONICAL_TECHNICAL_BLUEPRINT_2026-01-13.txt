Purpose
This document is the technical companion to “ASHBY — THE CANONICAL VISION”.
The Vision defines what Ashby MUST be.
This Blueprint defines how the architecture is expected to LOOK when implemented:
- stable seams (interfaces)
- stable directory responsibilities
- canonical runtime pipeline
- canonical data model fields (at a practical level)
- the minimum “rails” required before building more modules/agents

This is still human-readable. It is not raw code.
But it is specific enough that if two engineers build it independently, they converge.

============================================================
1) Canonical separation: WHAT vs HOW (enforced in code layout)
============================================================

Ashby must make the WHAT/HOW boundary physically obvious in the repository.

- WHAT lives in:
  - core policy
  - truth gate
  - state model
  - sessions
  - modules (domain interpretation)

- HOW lives in:
  - adapters (vendor/protocol calls)
  - transports (telegram/cli/http)
  - connectors (filesystem/network clients)
  - device SDK wrappers

If a file contains both WHAT and HOW, the architecture is leaking.

3) Canonical runtime pipeline (pseudo-logic)
============================================================

This is the “thin router” law expressed as steps.

router.handle(request_envelope):
  1) intent = nlu.extract_intent(request_envelope)
  2) target_set = registry.resolve_targets(intent)
  3) policy = policy_engine.evaluate(user_id, intent, target_set, profile, session_state)
     - returns ACT/CLARIFY/CONFIRM/REFUSE/DEFER/NO-OP
  4) if policy is CLARIFY/CONFIRM/REFUSE/NO-OP:
       return truth_gate.render(policy_payload, posture=UNKNOWN or NO-OP)
  5) module = module_registry.select(intent.intent_type)
  6) module_output = module.handle(intent, target_set, policy, session_state)
     - returns one of:
       - ActionPlan
       - ClarifyPayload
       - PipelineRunPlan (for compute modules)
  7) execute module_output via tool_runtime or pipeline executor
  8) update state model using ActionResults / verified telemetry
  9) response = truth_gate.render(results + state_updates + module_messages)
  10) audit log + structured logs
  11) return response

The router coordinates. It does not interpret.

============================================================

This section is not meant to replace Preservation Pack #2.
It is meant to prevent “field drift” when implementing.

4.1 IntentRecord (interfaces/models/intent.py)
Required:
- schema_name = "ashby.intent"
- schema_version
- intent_type (string)
- raw_text (string)
- user_id (string)
- correlation_id (string)
- requested_at (ISO8601 UTC)
- target_hints:
    - zones: [zone_key] (optional)
    - devices: [device_key] (optional)
    - entities: [entity_key] (optional)
- parameters: dict (canonical, typed where possible)
- ambiguity:
    - is_ambiguous: bool
    - ambiguity_reason: str | null
    - confidence: float 0..1

4.2 TargetSet (interfaces/models/intent.py)
Required:
- zones: [zone_id]
- devices: [device_key]
- entities: [entity_key]
- blast_radius:
    - device_count: int
    - zone_count: int

Required:
- decision: "ACT"|"CLARIFY"|"CONFIRM"|"REFUSE"|"DEFER"|"NO-OP"
- reason: str
- requires_confirmation: bool
- confirmation_payload: dict | null
- refusal_payload: dict | null
- clarify_payload: dict | null
- constraints:
    - allowed_tools: [tool_name] | null
    - max_blast_radius: int | null
    - profile: "LOCAL_ONLY"|"HYBRID"|"CLOUD"

4.4 ToolCall (interfaces/models/action.py)
Required:
- tool_name (string)

Required:

Required:

- lineage: { parent_run_id | null }

5) Truth Gate mechanics (platform-level)
============================================================

Truth Gate is not “tone control.”
It is the enforcement of Action-Gated Truth.

Truth Gate responsibilities:
- pick truth posture for the response based on ActionResults / state / artifacts
- block forbidden phrases that imply certainty without evidence
- force explicit representation of partial success
- force explicit admission of unknown/stale state
- ensure claims that reference reality link back to truth objects

Practical rules:
- If any target failed: response must mention partial success.
- If verification_class is LAST_KNOWN: response must include uncertainty marker + offer refresh.
- If state belief is stale: response must label staleness and propose verification.
- If a module produces artifacts: response may claim artifact existence VERIFIED if artifact files exist + hash.

============================================================
6) Policy + confirmation mechanics (safety rails)
============================================================

Policy is where “Ashby treats humans safely” becomes code.

Minimum confirmation triggers (examples, not exhaustive):
- blast_radius >= threshold
- dangerous device categories (locks, garage, heating extremes)
- ambiguous targets (multiple matches)
- profile escalation required (LOCAL_ONLY can’t run a requested cloud tool)
- repeated failures or contradictory state

Confirmation must be explicit and logged:
- what is being confirmed
- why it needs confirmation
- what will happen if confirmed
- scope of effect

============================================================
7) Scheduler/watchers mechanics (safe autonomy)
============================================================

Scheduler primitives:
- Job (id, schedule, scope, action_plan, constraints)
- Watcher (id, trigger, condition, action_plan, throttle, kill switch)

Watcher constraints:
- must never exceed configured blast radius
- must respect profiles and permissions
- must fail safe (no repeated runaway actions)
- must log every run with ActionResults

============================================================
8) Evaluation harness (how we prevent platform rot)
============================================================

The harness must:
- run the same pipeline as production transports
- test truth gate phrases and posture enforcement
- test partial success behavior
- test removability (disable a module, verify graceful degradation)
- test adapter swap (replace vendor adapter, verify modules still compile and run)

Golden scenarios should include:
- mixed success device operations
- stale state contradictions
- ambiguous target resolution requiring clarify/confirm
- offline profile enforcement

============================================================
9) “Reality drift” checklist (what to look for during development)
============================================================

If you see any of these patterns, stop and fix architecture before adding features:

- A module calls a vendor SDK directly
- An adapter returns only raw text without structured results
- Router grows with domain logic (“if intent == lights then …” everywhere)
- Policy decisions happen inside module code without platform enforcement
- Truth Gate is bypassed because “it’s annoying”
- State is inferred from chat history (“the user said earlier”)
- Config changes behavior without validation
- Adding a new module requires editing many unrelated files (boundary collapse)
