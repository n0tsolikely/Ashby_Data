SYSTEM CONTEXT STATEMENT
- Dungeon 11 requires a chat-authored template flow:
  - User requests a template in chat
  - Stuart proposes a draft (template.md + metadata)
  - User previews in UI and explicitly Save/Cancel
- We already have:
  - Chat API endpoints: `/api/chat` and `/api/chat/global`
  - Chat schema with action buttons (open_session, jump_to_segment)
- We need to extend chat actions to carry a template draft safely and let the user save it via the Templates API.

REPO / MONSTER ANCHORS
- Backend chat schema:
  - `ashby/modules/meetings/schemas/chat.py` (ChatActionV1 union)
- Backend chat commands:
  - `ashby/modules/meetings/chat/commands.py` (handle_command + /help output)
- Backend chat endpoints:
  - `ashby/interfaces/web/app.py` (handlers for /api/chat and /api/chat/global)
- Frontend chat UI:
  - `webapp/stuart_frontend/stuart_app/src/components/stuart/ChatInterface.jsx` (action buttons)
  - `webapp/stuart_frontend/stuart_app/src/pages/Stuart.jsx` (owns chat state + modals)
- Templates API:
  - `/api/templates` (QUEST_191)

CONSTRAINT SUMMARY
- Template draft must NOT be persisted unless user hits Save.
- Chat should not spam technical backend state; it should feel like an LLM.
- Keep it deterministic:
  - Command-based trigger is fine for v1 (no magic NLP required).
- Avoid breaking existing chat action schema.

ANTI-DUP PLAN
- `rg -n "ChatActionV1|open_session|jump_to_segment" Ashby_Engine/ashby/modules/meetings/schemas/chat.py`
- `rg -n "onChatAction" webapp/stuart_frontend/stuart_app/src/pages/Stuart.jsx`

DELIVERABLES
- Backend:
  1) New chat action type: `template_draft`
     - Fields: kind, mode, template_title, template_text, defaults
  2) New command: `/new_template` (or `/template new`)
     - Produces a template draft action
     - MVP: heuristic draft generation from description (LLM upgrade optional later)
  3) Update `/help` text to include new command
- Frontend:
  1) ChatInterface shows a button for template_draft action
  2) Clicking opens a modal preview (markdown render)
  3) Modal has Save/Cancel:
     - Save calls POST /api/templates (persist)
     - Cancel closes modal and discards draft
