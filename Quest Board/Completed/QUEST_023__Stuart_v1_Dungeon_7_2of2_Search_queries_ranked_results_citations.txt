DESCRIPTION
────────────────────────────────────────
Implement the query path that turns a user search intent (e.g., “find kimchi entry”) into deterministic ranked results with snippets and citations pointing back to transcript segments.

SCOPE / OBJECTIVE
────────────────────────────────────────
Implement:
- Query execution against SQLite FTS.
- Ranked result set with:
  - session pointers
  - snippets
  - citations/pointers back to transcript segments (segment_id/timestamps)

LLM usage rule (v1):
- LLM may be used after retrieval for intent parsing or summarizing top-N results.
- LLM must not brute-force scan filesystem for retrieval.
Deliver results in a structure doors can render (web/telegram).

OUT OF SCOPE
────────────────────────────────────────
Full export/download flows (handled in Dungeon 8).
Vector retrieval / embeddings.
Multi-user permissions.

DEPENDENCIES
────────────────────────────────────────
- QUEST_022

EXECUTION NOTES
────────────────────────────────────────
Search results must be citation-aware. If you can’t point back to transcript anchors, the result is not valid.
Keep results deterministic-first; LLM is presentation, not retrieval.

COMPLETION CRITERIA (REQUIRED)
────────────────────────────────────────
- [x] A query like “kimchi” returns ranked results quickly via SQLite FTS.
- [x] Results include snippets and at least one citation/pointer back to transcript anchors per result.
- [x] Result structure is door-agnostic (suitable for webapp rendering and minimal telegram output).
- [x] If no results, system returns a clear “no hits found” response without hallucinating sessions.
- [x] A minimal integration test shows search can be invoked from the router as a plan step (execution can be immediate).
