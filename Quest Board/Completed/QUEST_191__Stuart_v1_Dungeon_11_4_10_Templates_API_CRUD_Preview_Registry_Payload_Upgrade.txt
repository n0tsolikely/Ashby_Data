DELIVERABLES
- Backend routes:
  1) GET `/api/templates?mode=<meeting|journal>&q=<optional>&limit=&offset=`
  2) GET `/api/templates/{template_id}?mode=<...>&version=<optional>`
  3) GET `/api/templates/{template_id}/versions?mode=<...>`
  4) POST `/api/templates/draft` (preview-only; accepts text input, returns draft)
  5) POST `/api/templates` (persist; create new template or new version)
  6) DELETE `/api/templates/{template_id}?mode=<...>&confirm=true`
- Registry upgrade:
  - `/api/registry` includes `templates_by_mode` with template_id, template_title, template_version, source
- Tests:
  - API tests cover list/get/create/version/delete + registry templates_by_mode

TASK BREAKDOWN
1) Add API models (v1)
   - Prefer new file: `ashby/interfaces/web/templates_models_v1.py`
   - Models:
     - TemplateDescriptorV1 {template_id, template_title, template_version, mode, source}
     - TemplateRecordV1 {descriptor + defaults + created_at + updated_at + template_text}
     - TemplateDraftV1 (same shape as record but `persisted=false`, and no stable template_id until saved)
     - ListResponseV1 wrapper (or reuse PaginationV1 + data list)
   - Keep consistent with existing envelope patterns in api_models_v1.

2) Implement templates API router
   - Add: `ashby/interfaces/web/templates_api.py`
   - Use FastAPI APIRouter
   - Wire into app.py:
     - `app.include_router(templates_router, prefix="/api")` (or `/api/templates`)

3) Implement handlers using template_registry + TemplateStore
   - GET list:
     - use template_registry.allowed_templates_for_mode(mode) (now includes user templates)
   - GET record:
     - use template_registry.load_template_spec(...) + TemplateStore metadata to expose title/defaults/text
   - POST draft:
     - input: mode, template_title (optional), source_text
     - output: TemplateDraftV1 with generated template_text + defaults
     - IMPORTANT: does not write to disk
     - Implementation may be heuristic-only for now; LLM-assisted generation is introduced later (QUEST_195)
   - POST create:
     - input: mode, template_title, template_text, defaults, optional template_id (if creating new version)
     - create new template via TemplateStore (version=1) OR new version (version++)
     - return TemplateRecordV1
   - DELETE:
     - require confirm=true query param; otherwise return 400

4) Upgrade registry payload
   - Edit: `ashby/interfaces/web/registry_api.py`
   - Add `templates_by_mode` key:
     - mapping: mode -> list[TemplateDescriptorV1]
   - Ensure existing keys remain unchanged (back-compat for frontend).

5) Tests
   - Add: `tests/test_templates_api_crud_and_registry.py`
   - Use FastAPI TestClient pointed at app
   - Use temp STUART_ROOT env to isolate filesystem store
   - Assertions:
     - POST /api/templates creates template (v1)
     - GET list returns it under mode
     - GET versions returns [1]
     - POST create with same template_id creates v2
     - DELETE removes it
     - GET /api/registry includes templates_by_mode with titles
