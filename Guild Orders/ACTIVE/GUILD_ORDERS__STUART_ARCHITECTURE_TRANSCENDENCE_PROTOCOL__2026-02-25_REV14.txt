- UI label when unmapped is “Speaker-01”, “Speaker-02”, … (pretty display); internal IDs remain SPEAKER_01 etc.
- “Mentioned” ≠ “Attendee”:
  - Mentioned = keyword match in transcript text

   - speaker_map_id

   - updated_at
   - active scope: one ACTIVE mapping per session at a time (or a session pointer to “active transcript_version_id”)
2) Overlay seeding rule:
   - When a new transcript_version is created within the same session, seed its speaker_map from the most recent ACTIVE mapping (same session) by default.
   - Editing a mapping affects ONLY the selected transcript_version (unless the user explicitly selects a different transcript_version).
3) Endpoints:
   - GET/PUT speaker map for a transcript_version_id (session_id may resolve the current ACTIVE transcript_version)

     - Mentioned (transcript keyword)

DUNGEON 7 — FORMALIZATION MVP: END-TO-END REAL FORMALIZATION (NOT A TRANSCRIPT COPY)
======================================================================
Goal:
Make Stuart produce a legitimate formalization from a transcript using a SIMPLE DEFAULT TEMPLATE as a test, then expand.

User requirement (explicit):
- At minimum, Stuart must: audio → transcribe(+diarize) → formalize (meeting) using a default template.
- Today’s default template can be “Summarize this transcript with all important information in a structured organized doc” as a test.
- Journal formalization should be more near-verbatim, cleaned of filler/stutters.

Rules:
- Formalization is loss-minimized restructuring (not lossy “summary”).
- Must not fabricate facts.
- Must be template+retention governed.
- Transcript MUST be provided to the LLM as annotated speaker-turn segments: (segment_id, timestamps, speaker_label, mapped_name when available).
- Sections are content-driven by default (do not show empty sections unless explicitly enabled).
  - show_empty_sections is a control with TEMPLATE DEFAULT + UI OVERRIDE (1C). Default: false.
- Citations-in-document are optional and must NOT be coupled to retention.
  - include_citations is a control with TEMPLATE DEFAULT + UI OVERRIDE (2C). Default: false.
- Evidence maps are primarily for model/dev truth discipline; users do not need them front-and-center.

Tasks:
0. Template storage paths (canonical):
   - System templates (committed): Ashby_Engine/ashby/modules/meetings/templates/system/
     - meeting default: Ashby_Engine/ashby/modules/meetings/templates/system/meeting/default.md
     - journal default:  Ashby_Engine/ashby/modules/meetings/templates/system/journal/default.md
   - User templates (initial v1 store): Ashby_Engine/ashby/modules/meetings/templates/user/
     (Later this becomes a user data store + marketplace; for now this path is canonical.)

1. Ensure DEFAULT system templates exist and remain the baseline test templates:
   - Meeting Minutes (structured sections; include a final catch-all section like “Additional Notes”)
   - Journal Entry (near-verbatim cleaned narrative + key points)
   - Template authoring may include optional “always show these sections even if empty” intent via show_empty_sections defaults.

2. Wire formalization stage to:
   - accept transcript_version_id (canonical scope)
   - accept mode/template/retention/profile
   - accept show_empty_sections + include_citations controls (template defaults + UI overrides)
   - produce outputs (see Output artifacts)
   - LLM input MUST be annotated speaker-turn segments (see Rules) — not just raw transcript_text.
   - Template content MUST shape the prompt; no hard-coded provider prompt that ignores selected template.
   - Template shaping approach: BOTH schema/structure derived from template + raw template text (D3).
     - Schema drives canonical minutes.json/journal.json structure.
     - Raw template text is provided to the LLM to enforce formatting/field filling.

3. Evidence map (always generated, user-facing optional):
   - Evidence map must include segment references for all claims of fact, decisions, and action items.
   - Evidence map is ALWAYS generated for each formalization run (1A), regardless of include_citations.

Scope Model (binding):
A) SESSION scope (Hard Lock)
- SESSION scope means: ONLY use artifacts from the currently selected session in the left panel (selected_session_id).

  - references a session title/date/id explicitly, OR
  - clicks/selects a session in the left panel while GLOBAL is active.

  - scope_toggle: SESSION|GLOBAL

  - active_overlay_id / speaker_map info (if any)
  - active_mode/template/retention/profile selections (if present)

Objective outcomes:
- Session scope Q&A: answer questions only using the selected session’s artifacts.

Hard contracts (binding):

   - Global responses may return open_session actions for hits.
3) Planner/clarify behavior is allowed ONLY for explicit action commands:
   - /upload, /transcribe, /formalize, /export, /map_speakers, etc.

   - If evidence is missing, say so. No citations → label unknown or omit.
5) API consolidation:
   - /api/sessions is canonical session listing.
   - /api/library is TEMP only (alias/redirect to /api/sessions) and may be removed once legacy UI is gone.

- Chat history is ONE persistent thread (not per-session), even as the user switches sessions/tabs.
