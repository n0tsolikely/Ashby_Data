BUG A — Action-Gated Truth Violation (False state claims)

Symptom (from Telegram):
Ashby states “lights are off” / “lights are on” / “set to X%” without confirming via:

a successful device action result, OR

a live device state query.

Impact:
Ashby lies (unintentionally) and destroys trust. User corrections don’t reconcile internal state.

Fix:
Add a hard rule: No state assertions unless verified.

After an action: only speak in the past tense if the action returned success.

If no verification: speak as “last known state” or ask to confirm (“I don’t have a fresh state read — want me to check?”).

If user contradicts: mark internal state as stale/unknown until refreshed.

Acceptance test:
If Ashby did not execute+confirm or query state, it must never say “definitely on/off” — only “last known” or “I’m not sure yet.”

BUG B — HouseState authority / drift (state not treated as source of truth)

Symptom:
Chat layer improvises continuity (“it was off before we kicked it back up”) and doesn’t reconcile when the user reports reality (e.g., “it’s actually 100%”).

Fix:
Make HouseState the only source of truth for responses.

Update HouseState only from action results and/or explicit get_state() reads.

Add state_confidence (VERIFIED / LAST_KNOWN / UNKNOWN).

Response templates must respect confidence.

On user correction: set confidence to UNKNOWN and trigger a state refresh or ask.

Acceptance test:
When user says “it’s actually 100%,” Ashby replies like:
“Yeah — my state was stale. Want me to query Thor right now to sync?”

BUG C — “Again” / last_delta not applied correctly (repeat logic broken)

Symptom (7:12 PM):
User: “Again”
Ashby repeats same brightness instead of applying the last delta direction/step.

Fix:
Store last_delta per user session (already in design) and apply it when intent is repeat/again/more/less with no new magnitude.

If last_delta is None → default step (e.g., ±100 on 0–1000 scale).

If last action was absolute set (e.g., “set to 30%”) then again should repeat the absolute OR apply the last delta? (Pick one rule and make it consistent — I recommend: again repeats last command type. If last was a set, repeat set; if last was delta, repeat delta.)

Acceptance test:
“More” then “Again” continues moving the same direction by the same step.

BUG D — Command acknowledged but not executed (false success + execution failure not surfaced)

Your correction: He said it’s on, but it didn’t turn on even after multiple tries.

Likely causes (stacked):

Target resolution failure (Thor vs thor / alias normalization / group lookup mismatch)
