2) Windows + SD card triage (what we saw)
====================================================================
We pulled the Pi SD and put it in the laptop.
Windows Disk Management showed:
- A small FAT32 partition (“bootfs” ~512MB) visible in Explorer.
- A large second partition (ext4) present but not normally mountable by Windows.
We could browse boot partition files like cmdline.txt, config.txt, overlays, dtb files, etc.

Attempted to mount ext4 via WSL:
- We attempted:
  wsl.exe --mount \\.\PHYSICALDRIVE4 --partition 2 --type ext4 --read-only
- We hit multiple issues:
  - “Unknown command: --mount” occurred at one point due to invoking the wrong “wsl” binary (Wsman Shell tool, not wsl.exe).
  - WSL mount flags mismatch / unsupported argument: “Invalid command line argument: --read-only”
  - Then: “The system cannot find the drive specified. Error code 0x8007000f”
  - Diskpart showed Disk 4 Online 119GB; we tried “offline disk” and later “online disk” and saw:
    “Virtual Disk Service error: The object is not found.”

Important practical lesson:
- Diskpart state changes + some USB hubs can make devices disappear / stop enumerating temporarily.
- The Anker hub later stopped detecting SD readers properly (no connection sound, no drive), while other ports did work.

Decision:
- Stop burning time fighting Windows/WSL raw-disk mounting while streaming.
- Use the Pi itself (which can read ext4 natively) to do the backup safely.

3) Recovery plan chosen (the correct move)
====================================================================
Instead of risking more corruption:
- We booted the Pi normally (it did come up).
- We plugged a separate SD card into the Pi via SD-to-USB adapter.
- We backed up from the Pi’s internal SD (mmcblk0) → external USB SD (sda).

We validated devices with:
lsblk
and saw:
- mmcblk0p1 mounted at /boot/firmware
- mmcblk0p2 mounted at /
- External SD appeared as sda1 mounted at /media/notsolikely/SD

====================================================================
4) Backup execution (what worked, what failed, and why)
====================================================================
First attempt (copy full home dir):
cp -r /home/notsolikely /media/notsolikely/SD/pi_backup

This failed with:
- “cannot create symbolic link … Operation not permitted”
because the destination filesystem (likely exFAT/FAT) does not support Linux symlinks/permissions the same way.

Correct solution:
- Use tar to preserve structure in an archive (and avoid direct symlink creation issues):
tar -czf /media/notsolikely/SD/pi_backup.tar.gz /home/notsolikely

Result:
- pi_backup.tar.gz created successfully.
- Size observed: ~411MB (later shown around ~420MB on Windows).
- We also had a “pi_backup/” folder created from earlier attempts, but the tar.gz is the real reliable artifact.

We then unmounted safely and noted the SD got warm (normal under write load).

8) Python environment fixes (the actual blocker)
====================================================================
We tried running:
python3 ash.py
and got:
ModuleNotFoundError: No module named 'openai'

We tried pip and got:
“pip: command not found” and “externally-managed-environment” errors (PEP 668).
That’s Debian/Ubuntu protecting system Python from random pip installs.

Correct fix (what we did):
- Create a NEW venv inside the project directory:
python3 -m venv .venv
source .venv/bin/activate

- Upgrade pip inside the venv:
python3 -m pip install --upgrade pip

- Install required packages inside the venv (NOT system-wide):
python3 -m pip install openai tuya-iot-py-sdk

This succeeded and installed all dependencies cleanly.

Next blocker:
Running ashpi_test.py then failed because:
OPENAI_API_KEY environment variable was not set.

Fix:
- Export key in the shell before running, or load it via env file logic.
(You did the smart thing and hid the key from stream.)

10) Bugs / behavior issues discovered during live testing
====================================================================
BUG 1 — “Turn on to 10%” while light is OFF doesn’t actually turn it on first
Observed behavior:
- Command: “turn him on to 10%”
- Ashby acted like it did it, but the light stayed off (because it didn’t issue an ON + brightness sequence).
Expected:
- If light is OFF and user says “on to X%”, system must:
  1) turn on
  2) then set brightness to X
Or do brightness set in a way that implies ON for that device API.
This needs an explicit “ensure_on” in the action path when an absolute brightness command is used on an OFF device.

BUG 2 — “again” intent causes a lighting loop / doesn’t apply last_delta correctly
Observed:
- After reaching 40%, saying “again” triggered GPT intent classification (“again”) and it responded:
  “Cranking Thor up to 40% again—oh wait, we're already there!”
It did NOT apply +10% step (or last_delta inheritance) properly.
Expected:
- “again” should inherit last direction and apply another step (e.g., +100/1000) even if the exact level repeats.
- Also: It should not announce doing something it already knows is redundant.

BUG 3 — “yes” intent acknowledged a change but state reported inconsistently
Observed:
- Ashby said it was bringing Thor to 50%, but later “what’s he at” reported 40%.
This suggests:
- Either the 50% action didn’t execute,
- or the state tracker didn’t update,
- or the answer came from stale / assumed state rather than verified HouseState.

Meta-note on logs:
- When the log shows “[AshPi INFO] Setting group…” without “[AshPi NLU] GPT intent called…”, that usually means it matched a local rule/intent path (fast path) instead of GPT NLU.
- When it shows “[AshPi NLU] GPT intent called…”, it went through GPT intent resolution.
