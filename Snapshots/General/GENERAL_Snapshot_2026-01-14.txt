3) Stuart: the final agreed build target
Stuart is a deterministic, artifact-first pipeline for “meetings” that produces evidence-backed outputs.

3.1 Identity / placement
- Module name (user-facing): Stuart
- Module key (internal): meetings
- Runs as an Ashby module, not as a separate engine.

3.2 Pipeline sequencing (first working vertical slice)
Agreed build order (CLI-first):
1) Scaffold (folders + interfaces + minimal wiring)
2) Stage 0: session init + ingest (deterministic storage + manifests)
3) Stage 1: normalize audio (ffmpeg/local)
4) Stage 3: transcribe (ASR adapter)
5) Captions: SRT output
Then later (after working slice):
- diarization, alignment, formalization, indexing/retrieval expansions

3.3 Storage and artifacts (non-negotiable)
- Artifacts are the canonical memory.
- Deterministic paths under runtime/stuart/sessions/{session_id}/...
- No silent overwrite of ground truth artifacts.
- New execution = new run_id (append-only behavior).

3.4 Truth/evidence discipline
- Any “claim” (decisions/actions/incidents/attribution) must include citations into transcript segments/timestamps.
- Speaker labels (SPEAKER_00, etc) are ground truth; real names are overlays only.
- If evidence is weak, output must label ambiguity instead of fabricating certainty.

3.5 Profiles/adapters
- LOCAL_ONLY default, with HYBRID/CLOUD as explicit choices requiring disclosure+consent logging.
- Stuart has domain-level adapters (ASR, diarization, formalization, caption renderers, indexing), while Ashby has platform-level adapters (devices/vendors/storage/transports).

8) New-chat handoff workflows agreed in this chat
We converged on a strict method to avoid “what is this file?” tangents and prevent drift:

A) Always send a directive first
- First message in a new chat must state: objective, what the uploads represent, and that output must be stepwise commands + file contents + smoke tests only.

B) Use a checkpoint before coding
- New chat must list received filenames and name exact repo paths it will touch before writing any code.

C) Upload strategy (10-file limit)
- Preferred pattern: two-upload flow:
  1) Engine.zip + Data.zip + Batch 0 runbook/contracts (and optional blueprint) in one upload.
  2) Optional extra context file(s) (e.g., canonical vision) in the next upload.

D) Two-chat separation (to keep threads clean and non-drifting)
- Chat A: Ashby Batch 0 implementation only (platform primitives).
- Chat B: Stuart implementation only (module build using Stuart Execution Pack).

9) Implementation workflow agreed (brains/hands)
- User works locally (WSL/Ubuntu + VS Code recommended).
- Assistant outputs in batches to avoid truncation:
  - FILES AFFECTED
  - full file contents in code blocks
  - commands to create/move files
  - smoke tests to run after each batch
- The user runs commands and pastes terminal output back for patching.

Commit strategy:
- Build locally first; do not push noisy half-baked commits.
- Once the vertical slice runs, then create a clean canonical commit.
