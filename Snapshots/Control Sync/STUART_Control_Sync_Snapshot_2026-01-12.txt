3) MATERIAL DECISIONS LOCKED IN THIS CONTROL SYNC
===============================================================================

3.1 Interface decision
- Stuart’s primary interface is a local-first web app (PWA-capable).
- The UI must support: mode selection, template selection (scoped by mode), retention selection,
  transcript visibility toggles, and explicit execution gating.

3.2 Explicit execution gating is default
- Uploading a file does not automatically run the pipeline unless explicitly triggered.
- This prevents wrong-file accidents and privacy/visibility failures.

3.3 “Memory” is artifacts + index, not chat memory
- Recall is achieved by storing session artifacts and indexing them for query.
- Users can ask: “pull up all meetings mentioning X,” “all incidents with Y,” etc.

3.4 Modes vs templates separation
- Modes define output type and governance; templates define formatting + section schema.
- Templates are scoped by mode; template selection UI is contextual.

3.5 Retention knob is a first-class control
- LOW/MED/HIGH/NEAR-VERBATIM modifies how much phrasing is preserved in formalizations.
- Transcript remains verbatim; retention knob affects formalized outputs.

3.6 Evidence posture (anti-hallucination)
- Formalizations must cite evidence back into the aligned transcript.
- If evidence is missing or ambiguous, Stuart flags uncertainty rather than inferring.

3.7 Telegram integration parked
- Telegram is not needed for core product value and introduces LOCAL_ONLY truth issues.
- Telegram may return later only as a convenience door; not a core path.

3.8 Canon separation doctrine
- Canonical Vision is separate from Codex.
- Codex remains non-summarized statute; snapshots/roadmaps are allowed but must not replace it.

===============================================================================
4) KEY RISKS / CONCERNS IDENTIFIED (AND HOW WE HANDLED THEM)
===============================================================================

4.1 Long sessions and output limits
- Risk: very long transcripts + formalizations exceed single-output limits.
- Mitigation: hierarchical chunking + staged formalization + multi-artifact export formats.

4.2 Speaker diarization realities
- Risk: diarization isn’t perfect; overlap exists; misattribution creates trust collapse.
- Mitigation: conservative attribution defaults, explicit speaker mapping overlays, evidence anchors.

4.3 Privacy / offline truth
- Risk: networked chat frontends compromise “LOCAL_ONLY” at the upload path.
- Mitigation: web app + local service is canonical; profiles disclose any remote execution.

4.4 Template ecosystem risks
- Risk: template sharing/marketplace leaks PII and introduces injection vectors.
- Mitigation: template sharing is future-gated; scrubbers + review + provenance required.
