FORMALIZATION (Stuart definition; binding)
- A formalization is NOT “a file containing metadata fields.”
- A formalization is a LOSS-MINIMIZED semantic restructuring of a complete transcript into a decision-complete structured artifact.
- It is “meeting minutes / journal / incident report / etc.” generated from the transcript and organized for usability.
- It is “summarization” only in the sense of removing conversational noise and reorganizing by semantic role, NOT in the sense of losing materially relevant information.
- Templates control STRUCTURE + REQUIRED FIELDS + REQUIRED COVERAGE, not “how much we feel like summarizing.”
- Retention level controls verbosity / quoting vs paraphrase, but must preserve all materially relevant context.
- Formalization output formats may include Markdown + PDF render + JSON pack, but the “formalization” is the semantic artifact itself, not the storage wrapper.

NON-NEGOTIABLE:
- No fabricated transcript content.
- No fabricated decisions/actions/quotes in formalization.
- Evidence map must anchor claims to transcript segments (segment_id + timestamp ranges).

SECTION 13 — PIPELINE STAGING (CRITICAL CORRECTION)
======================================================================

BINDING STAGING RULE:
- DIARIZATION belongs to TRANSCRIPTION, NOT FORMALIZATION.

WHY:
- Diarization changes the structure of the transcript (speaker-separated segments).
- Formalization consumes a transcript and re-structures it into a usable artifact; it should not be responsible for speaker separation.

Therefore we enforce a 2-stage user action model:

STAGE A: TRANSCRIBE (with optional diarization)
- Inputs: audio, ASR config, diarization toggle
- Outputs:
  • transcript segments (segment_id, start_ms, end_ms, text)
  • optional speaker labels (SPEAKER_01, SPEAKER_02, etc.)
  • optional confidences (if available)
- Artifact must be stored as a transcript version (see Section 14).

STAGE B: FORMALIZE (uses an already-created transcript version)
- Inputs:
  • selected transcript version (from Stage A)
  • mode (minutes/journal/incident/etc.)
  • template (structure contract)
  • retention level (verbosity/quoting policy)
  • profile (local/hybrid/cloud routing)
- Outputs:
  • formalization artifact (semantic layer) rendered to md/pdf and/or JSON packs
  • evidence map anchored to transcript segments + timestamps
  • run logs + manifest

THIS IS NON-NEGOTIABLE:
- The UI must not imply that “formalize” produces diarization.
- The UI must not imply that “transcribe” is fake or LLM-generated.
- The UI must not hide stage boundaries.

SECTION 14 — TRANSCRIPT VERSIONING (REQUIRED FOR TRUST + RERUNS)
======================================================================

A session must support MULTIPLE transcripts, because the user can legitimately want:
- a fast transcript (no diarization)
- a diarized transcript (slower, but speaker-separated)
- a higher-quality ASR transcript (different model/config)
- a corrected/edited transcript overlay (future)

Therefore we introduce the concept:

TRANSCRIPT_VERSION
- transcript_version_id (stable)
- created_at
- diarization_enabled: true/false
- asr_engine: (local whisper/faster-whisper/remote/etc.)  [implementation detail; can start as “default”]
- segments: list of (segment_id, start_ms, end_ms, speaker_label?, text, confidence?)
- overlay_speaker_map_id (optional reference; see Section 15)

UI REQUIREMENT:
- Transcript selector dropdown in the Session tab:
  • “Transcript v1 (no diarization)”
  • “Transcript v2 (diarization)”
  • etc.
- The formalization config must reference WHICH transcript version is being used, so reruns are explainable.

Without transcript versioning, you get “why did the minutes change?” confusion and trust collapses. That’s bullshit—we avoid it.

SECTION 15 — SPEAKER MAPPING (OVERLAY, NOT TRUTH MUTATION)
======================================================================

Speaker labeling must be handled as an overlay layer:

DIARIZATION OUTPUT:
- speaker labels like SPEAKER_01, SPEAKER_02 (ground truth-ish)
- these are stable identifiers for segments

USER MAPPING OVERLAY:
- SPEAKER_01 -> “Greg”
- SPEAKER_02 -> “Peter”
- etc.

RULE:
- Changing the mapping does NOT rewrite transcript text; it changes how we DISPLAY and how CHAT/FORMALIZATION interpret speakers.
- This mapping must be versioned (or at least tracked) so outputs can declare which mapping was used.

WHY:
- It enables cross-session queries like “meetings Greg was at” without hallucinating identities.
- It allows corrections (speaker overlay edits) after diarization.

CHAT + FORMALIZATION must reference the speaker map used, or at minimum be consistent with current mapping and show evidence citations.
