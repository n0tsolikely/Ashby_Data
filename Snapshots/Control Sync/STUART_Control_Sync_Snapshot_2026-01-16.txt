4) STUART AS A MODULE (KEY DESIGN DECISIONS)
──────────────────────────────────────────────────────────────────────────────
4.1 Stuart is a module, not baked into Ashby core
We aligned on the correct separation:
- Ashby system router should only dispatch to modules (meetings module, inventory module, etc.).
- Stuart’s domain logic must live inside its module. If Stuart is removed, Ashby should not have “dead Stuart logic” stuck in the router.

4.2 Two-layer routing model
We clarified the important architecture:
- Platform routing (Ashby-level): decide which module handles the user request.
- Module routing (Stuart/meetings-level): within the meetings domain, interpret intent (“search entries”, “formalize”, “edit template”, etc.)

This prevents Ashby core from becoming a god-object and keeps modules removable.

4.3 Module naming decision
Decision: internal module key remains “meetings” (domain-based), while “Stuart” is the persona/UI name.
Rationale:
- Domain-based keys scale better than persona-based keys.
- Keeps alignment with existing spec and avoids renaming churn.

(Alternative considered: “records/logs” as broader domain name, but we stayed with meetings for v1 to remain consistent with existing canon.)

4.4 Where Stuart will live
Canonical location: ashby/modules/meetings/
We add this module zone (additive change) rather than moving old Ashby code around.

5) USER EXPERIENCE CONTRACT FOR STUART V1
──────────────────────────────────────────────────────────────────────────────
The biggest product decision of this Control Sync:
Stuart v1 must not feel like “CLI commands and paths.” It must feel like a chat-first assistant (Ashby-like) with file drop, consistent behavior, and minimal learning curve.

5.1 Primary interface: local webapp (ChatGPT-like)
Decision: Webapp is the primary interface for v1.
Expected UX:
- Chat pane + session history (like ChatGPT).
- User can select Mode + Speakers + Template in UI.
- User can also simply speak naturally: “Formalize this as meeting mode.”
- Chat commands must override UI state (and UI must update to match) to avoid “UI/backend mismatch” bugs.

5.2 Telegram is a thin “door,” not the foundation
Decision: Telegram is optional/secondary and exists mainly for:
- remote demo
- remote journaling/meeting capture from phone
It must not drive product complexity or force weird grammar.

5.3 Telegram interaction model (buttons)
Decision: Telegram bot will use inline tappable buttons (not text command grammar), with a simple two-step flow:
- Step 1: Mode selection (Journal/Diary vs Meeting)
- Step 2: Speakers selection (Auto / 1 / 2 / 3+)
Then it runs DEFAULT TEMPLATE ONLY and returns the PDF as a Telegram attachment.

Template selection via Telegram is explicitly deferred (keeps Telegram dead simple).

5.4 Speaker controls: user-friendly, no “diarization” term
Decision: users select speakers in plain terms:
- Just me (1)
- Auto
- 2
- 3+
Defaults:
- Journal defaults to 1 speaker.
- Meeting defaults to Auto.
Users picking the wrong number is acceptable user error; we are not building “hero correction logic” for that.

5.5 Execution gating / confirmation rules
Key behavioral law:
- Upload/drop ≠ run.
- Stuart must have an explicit “go” signal before heavy processing.
Go signals include:
- user explicitly instructing in chat (mode/template/speakers specified), or
- user confirming a plan in the webapp, or
- user pressing Telegram buttons.

Plan preview requirement:
- If any part is ambiguous or comes from UI state that could be wrong, Stuart must plan-preview and require confirmation.
- If the user fully specifies everything in the message, Stuart can proceed without a second confirm.

5.6 Transcript handling
Decision:
- Transcript ALWAYS exists (ground truth).
- Visibility can be toggled in webapp to avoid forcing transcripts into every UX, but user must be able to access it.

5.7 Downloads (no folder spelunking)
Decision:
- User must be able to download outputs directly from the interface.
- Webapp provides download buttons (PDF/MD/transcript as allowed).
- Telegram returns artifacts as attachments.

6) STUART ENGINE SHAPE (PIPELINE, ARTIFACTS, OVERLAYS, SEARCH)
──────────────────────────────────────────────────────────────────────────────
6.1 Artifact-first storage
Decision:
- Stuart stores canonical artifacts under a dedicated root (STUART_ROOT; default under runtime/stuart/).
- Source media is stored internally for later reprocessing and retrieval.
- For video uploads: extract and store canonical audio for processing; keep original when feasible.

6.2 “One transcript, many formalizations”
Decision:
- A single ingested contribution produces a transcript artifact.
- Multiple formalizations (different modes/templates later) reference that transcript without duplicating work.
- Re-formalize later using the same stored artifacts; do not create a new session for each new formalization.

6.3 Speaker mapping is a required capability
Decision:
- Initial outputs may use SPEAKER_00 / SPEAKER_01.
- Users must be able to rename speakers (“SPEAKER_00 is Greg”) and have outputs re-render with names.
- “Only Greg said” type extracts require speaker mapping.

Implementation principle:
- Speaker naming is an overlay on top of ground truth, not destructive edits.

6.4 Minimal search/index layer is required in v1
Decision:
- We will not rely on the LLM scanning files.
- We implement deterministic retrieval first (SQLite FTS minimum) that returns hits + snippets + citations.
- LLM is used for intent parsing and presentation/rerank, not brute force.

Example expectation:
- “Find the journal entry where I talked about kimchi” returns matching sessions/snippets quickly and lets user export outputs.

6.5 Door abstraction for clarifications
Decision:
- Engine emits a door-agnostic “clarify prompt” structure (question + options).
- Telegram renders it as buttons.
- This design is meant to be reusable for future doors (Discord, etc.) without rewriting engine logic.
