Big takeaways from Ashby Codex (the rules that matter for Stuart)
1) Ashby is the platform “body”; Stuart is an “agent/organ”
   - Stuart should not become a separate forked platform with its own rules.
   - It should run on Ashby’s shared infrastructure: routing, truth gate, storage/IO patterns, state discipline.

2) Offline-first is a platform routing law, not a one-off feature
   - “Local-only” is a mode/profile (strict privacy).
   - “Cloud-enabled” is another mode/profile (for customers who don’t care).
   - The system must be designed so adding cloud later is NOT a refactor — it’s an adapter swap.

3) Adapters are the swap mechanism (model swapping without pipeline rewrite)
   - Transcription, diarization, summarization/LLM, and search/indexing should each be “capabilities” with adapter implementations.
   - The pipeline calls the capability interface; the router chooses the adapter.

4) Truth + Evidence are non-negotiable
   - For Stuart: “no hallucinations” means every extracted item (decision/action/etc.) must be backed by transcript evidence (timestamps/segment IDs).
   - If evidence isn’t present, Stuart must say “not in the recording” and shut up.
