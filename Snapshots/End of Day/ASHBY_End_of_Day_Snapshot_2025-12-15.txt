0) THE GOAL (locked)
Ashby should feel like a real person in the room who can control the house:
- Direct commands (“turn off thor”, “set to 50%”, “off”) => short, human replies + immediate execution
- Vibe statements (“it’s bright in here”, “that’s way too bright”, “damn it’s dark”) => chatty, contextual replies + either:
  - act if context is clearly lights (e.g., active lighting session / “in here”), OR
  - ask a human clarification about the subject if it’s ambiguous (“bright out today” => weather, not lights)
- Ashby must never *claim* an action occurred if router did not execute it (no “yeah it’s off” while lights are still on).

3) WHAT THE TESTS SHOWED (PROBLEMS OBSERVED)
A) Execution vs speech mismatch (critical)
- Ashby sometimes *said* lights were off/dimmed without the device actually changing.
Root cause:
- Router returns safe_chat() when target_group is missing, so the LLM “confirms” an action that never happened.
Fix direction:
- Router must only describe post-action reality; if no target, ask/clarify, don’t claim execution.

B) Target resolution is incomplete
- “Turn off the lights” often fails unless the user explicitly says “thor” / “captain”.
Root cause:
- group=None + no active session => target_group resolves to None => chat fallback.
Fix direction:
- Add a concept of “default group for this user/room” OR prompt for target.
- When a lighting session is active, interpret “the lights” as the active group.

C) Direction inheritance / emphasis scaling not right yet
- “way more down” sometimes only changed by 10% (step) instead of a big jump.
- After dimming, saying “more” sometimes went UP instead of “more down”.
Root cause:
- last_delta / direction inheritance logic isn’t consistently applied; “more” defaults to brighter unless tied to last direction.
Fix direction:
- Persist and reuse direction context: last_delta sign should decide what “more” means.
- Add “emphasis -> magnitude” mapping: “way/way way/retinas” => bigger delta; “a little” => small delta.

D) Reply length/style is not yet consistent
- Direct commands still sometimes produce long multi-sentence replies.
Fix direction:
- Implement reply-style routing:
  - if command-like => short
  - if vibe-like => chatty
This should be enforced inside lights.on/off/adjust returns (not left to random chat).

4) ARCHITECTURE CONCERN (IMPORTANT, CALLED OUT)
router.py is absorbing too much “brain” logic.
Current reality:
- Router is doing routing + policy + heuristics + style + clarification.
Scaling risk:
- Expanding to plugs/appliances/garage/cameras will turn router into a monster file.
Planned fix:
- Keep router as an orchestrator; extract “brain policies” into dedicated modules:
  - brain/policy_light_vibes.py (vibe phrase interpretation & clarifiers)
  - brain/reply_style.py (short vs chatty decision)
  - brain/target_resolution.py (resolve “the lights” -> default group / active session)
  - brain/direction_inheritance.py (more/again semantics + emphasis scaling)
Router should call these helpers, not own all the logic.
