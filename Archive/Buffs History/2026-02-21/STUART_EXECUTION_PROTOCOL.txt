3) DEFINITIONS (TERMS USED IN THIS PROTOCOL)
================================================================================
- Control Sync: decision forum. Produces binding Guild Orders and/or governance changes.
- Guild Orders: ordered scope list for the current Raid. Defines “done.”
- Raid: executing Guild Orders across one or more Dailies loops.
- Quest: atomic commitment. One quest = one cohesive deliverable capability/change.
- Quest Board: filesystem state machine (BOARD → ACCEPTED → COMPLETED).
- Snapshot: ground-truth receipt. What happened, what changed, what is now true.
- Execution Pack: time-bound runbook/contract for deterministic implementation.
- Canon: Codex + visions + governance rules; the law that prevents drift.
- Talent: new durable capability (not effort). See Talent Tree Guide.

Terminology hardlock (Stuart-specific):
- Transcript = produced from REAL audio by ASR (no fabrication).
- Diarization = speaker-separation metadata (optional at transcription stage).
- Formalization = structured “no-context-loss summary” produced FROM a transcript using a template.
  (The word “summary” is hated because it implies loss. Formalization must not lose meaning.)

APPENDIX A — PLATFORM TRUTH + PROFILE LOCK (BINDING ARCHITECTURE CONSTRAINTS)
================================================================================
NOTE:
- This appendix was previously a standalone buff in older sets.
- Governance requires a canonical buff set of THREE files.
- Therefore, the lock lives INSIDE this Execution Protocol to avoid a 4th file.

A0) WHAT THIS IS
- NOT a snapshot.
- NOT an execution log.
- A binding architecture lock so Stuart can scale without major refactors.

A1) TRUTH ARCHITECTURE (BINDING)
A1.1 Single Judge (Platform)
- Stuart has ONE platform-level truth judge: TruthGateJudge.
- The judge is reusable across modules.
- Truth law enforcement must converge here over time.

A1.2 Module Policies (Specialization)
- Each module provides module-specific TruthPolicy implementations (e.g., MeetingsTruthPolicy).
- Policies define what is considered a valid claim in that domain.

A1.3 Module-Local Checks (Allowed, but limited)
- Modules MAY keep local validation guards ONLY for cheap invariants/internal consistency:
  - citation segment_id exists
  - timestamps monotonic
  - speaker_id references are valid
- Modules MUST NOT implement a parallel “truth gate” deciding real-world truth.
  If the rule is about truth (decisions, actions, facts, attendance), it belongs in:
  Platform Judge + Module Policy.

A2) EVIDENCE + CITATIONS (BINDING)
- Claims must be evidence-backed.
- Citations must reference resolvable evidence anchors (segments / timestamps / artifact pointers).
- Output formats may vary (MD/PDF/JSON), but evidence semantics must be consistent.

A3) EXECUTION PROFILES + CONSENT (BINDING)
A3.1 “Profile” is NOT personality
- Execution profile = runtime constraints + egress policy (LOCAL_ONLY / HYBRID / CLOUD).
- Personality/tone = templates/prompts/module UX.
- These are separate concerns and must not be conflated.

A3.2 Single Execution Profile Substrate (Platform)
- Stuart must have ONE execution profile substrate.
- Modules must NOT maintain a parallel execution profile system.
- Canon substrate target: ashby.core.profile (and associated consent/egress logic).

A4) ROUTING + MODULE BOUNDARIES (BINDING)
A4.1 Thin Core Router
- Core router routes high-level intent to modules.
- Core router must NOT contain domain logic for specific modules.

A4.2 Module Routers
- Modules own their internal intent routing (module router).
- Removing a module must not leave dead module logic in core.

A4.3 Removability
- Modules must remain removable without breaking the platform.
- Platform provides shared primitives (truth judge, storage, results, profiles).
- Modules provide domain logic + policies + artifacts.

A5) FILE DISCIPLINE (BINDING)
- Big files are OK when cohesive (one capability, one reason to change).
- God files are NOT OK (mixed responsibilities/refactor bait).
- Prefer separated capabilities that call each other.
- Keep things together only when separation would be worse.

A6) GROWTH MINDSET (BINDING)
- Stuart v1 is starting scope, not end state.
- Build extension points (registries, interfaces, policy hooks) where natural.
- Prefer additive evolution over invasive refactors.

APPENDIX C — CURRENT RAID INVARIANTS (STUART v2 UI + LLM WIRING)
================================================================================
This appendix is a “don’t be dumb tomorrow” lock.
It does NOT replace Guild Orders or Quests; it prevents re-litigating decisions.

C1) Chat-first UX is the default
- On app open, Chat is the primary surface.
- Buttons exist as convenience accelerators for chat-commands.
- Every button must correspond to an equivalent chat command.

C2) No fake transcripts, no fake formalizations
- Transcript must be produced from REAL audio by ASR.
- Formalization must be produced from REAL transcript using template + retention settings.
- No “demo outputs” that look correct but are fabricated.

C3) Transcript versions are first-class
- We must support transcript variants (e.g., diarize ON vs OFF).
- Each transcript version must have a stable ID and be selectable in UI/API.
- Formalizations must reference transcript_version_id explicitly.

C4) Diarization belongs to TRANSCRIBE stage
- Diarization is controlled during transcription (toggle on/off).
- Formalization consumes transcript (with or without diarization); it does not “do diarization.”

C5) Base44 naming contamination is NOT allowed in backend
- UI can be ported from Base44, but engine naming stays Stuart/Ashby canonical.
- Temporary shims are allowed ONLY if the quests explicitly allow them.
- Target is clean naming + explicit stuartClient API contracts.

C6) Export bundles are a core product surface
- Export should bundle session artifacts (audio, transcripts, diarization, evidence, formalizations).
- Export filtering (transcripts only, formalizations only, full) must be supported.
