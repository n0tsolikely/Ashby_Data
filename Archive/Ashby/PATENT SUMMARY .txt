# ASHBY – STATEFUL AGENTIC HOME CONTROL PIPELINE

**Invention Summary (Patent Draft – Plain English)**

## 1. Technical Field

This invention relates to **intelligent control systems**, specifically **state-aware, agentic home automation systems** that integrate natural language understanding (NLU), execution routing, and large language model (LLM) reasoning **without allowing hallucinated or false system behavior**.

---

## 2. Problem Being Solved (This is key)

Current consumer assistants (Amazon Alexa, Google Assistant, Apple Siri):

* Treat commands as **stateless transactions**
* Confirm actions **without verifying execution**
* Have **no persistent world model**
* Can deny control authority (“I can’t do that”) even after performing actions
* Lose context immediately after each command
* Cannot reason about *what just happened* in the physical environment

They behave like **voice remotes**, not operators.

This causes:

* False confirmations
* Contradictory responses
* Broken conversational flow
* No learning from outcomes
* No internal accountability for actions taken

---

## 3. Core Idea (High-Level)

Ashby introduces a **stateful agentic control pipeline** where:

> **No language response is trusted unless it is grounded in verified system actions and injected system state.**

The system does not “chat about the house.”
It **operates the house**, then **updates its own internal world model**, and **speaks only from that updated reality**.

---

## 4. The Ashby Pipeline (The Invention)

### Step-by-Step Flow

1. **Natural Language Input**

   * User provides free-form speech/text (e.g., “turn it on”, “more”, “it’s bright”)

2. **Hybrid NLU Extraction**

   * Local rules + LLM-assisted NLU extract a structured intent:

     * action type (on/off/adjust)
     * target (explicit or implicit)
     * modifiers (brightness, delta, mode)
   * Supports **incomplete commands** (“turn it on”) using session carryover

3. **Intent Router with Session Awareness**

   * Router decides whether the input is:

     * an action
     * a follow-up
     * a clarification
     * or pure chat
   * Maintains **active sessions** (e.g., current lights being adjusted)

4. **Verified Action Execution**

   * Router calls real device control APIs
   * Actions are executed *before* any conversational response is finalized

5. **State Injection (Critical Novelty)**

   * After execution, the system injects a **factual system-state update** into the LLM context, e.g.:

     > “SYSTEM STATE UPDATE: Living room lights set to 60%.”

   * This update is **not user-visible**

   * It updates the assistant’s internal world model

6. **Hallucination-Guarded Chat Layer**

   * All conversational responses pass through a **safe chat wrapper**
   * Responses that contradict system agency are blocked or rewritten:

     * “I can’t control that”
     * “I don’t have access”
     * “As an AI…”
   * This layer **never blocks real actions**, only false narration

7. **Action Echo → Memory Loop**

   * Executed actions update:

     * short-term session state
     * long-term preference memory (e.g., preferred brightness)
   * Future decisions reuse this data automatically

8. **Grounded Response Generation**

   * The final response is generated **only after**:

     * action execution
     * state injection
     * hallucination screening
   * The assistant speaks **from reality**, not speculation

## 5. What Makes This Different (Patent-Relevant)

### 🔥 This is the money section

Ashby is **not** just:

* NLU
* Not just LLM chat
* Not just smart-home control

It is a **closed-loop, state-validated agent**.

Key differentiators:

1. **Action-First Truth Model**

   * The system only “knows” what it has actually done
   * No speculative confirmations

2. **System-Injected Reality**

   * State is injected as authoritative system context
   * Prevents conversational drift and hallucination

3. **Agency Enforcement**

   * The assistant is prevented from denying its own control authority
   * Unlike existing assistants, it cannot “forget” it runs the system

4. **Session-Based Natural Language Resolution**

   * Handles incomplete and follow-up commands naturally:

     * “Turn it on” → “Which lights?” → “Thor”
     * “More” inherits previous direction and magnitude

5. **Memory Based on Outcomes, Not Commands**

   * Preferences are learned from **successful states**, not raw instructions

6. **World Model over Command Model**

   * The assistant maintains a continuously updated internal model of the environment
