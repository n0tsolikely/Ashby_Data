2.3 Tuya Outlet Control – First Plug

Core script: **`ashpi_plug_test.py`** (you also have a saved reference like `ashpiplug.txt`)

Behavior:

- Imports Tuya client and env values.
- Connects using:
    openapi = TuyaOpenAPI(ENDPOINT, ACCESS_ID, ACCESS_KEY)
    openapi.connect(USERNAME, PASSWORD, "1", "smartlife")

- Defines `set_plug(on: bool)`:
  - Sends a command to:
    `/v1.0/devices/{DEVICE_ID}/commands`
  - Payload:
    - code: "switch_1"
    - value: True / False

- Basic test loop:
  - Press Enter → turns plug ON
  - Press Enter again → turns plug OFF

This is your **first confirmed working Tuya control** and is still the simplest
way to manually flip the plug from the Pi.


2.4 Tuya Lights Control – Bulbs

You saved a detailed file: **`ashpi_tuya_lights_setup.txt`** that documents how
we structured Tuya lights (Captain America, Thor, Sky).

Core concepts:

- A file like `ashpi_lights_tuya.py` that:
  - Uses Tuya OpenAPI to control bulbs.
  - Controls:
    - `switch_led`       (on/off)
    - `bright_value_v2`  (0–1000 brightness)
  - Keeps a small in-memory brightness cache to allow step up/down commands.

- Functions we discussed/used:

  - `step_brightness(target, delta)`
  - `turn_on(target, value=None)`
  - `turn_off(target)`

- `target` can be:
  - A specific bulb: `"thor_bulb"`
  - A group (later, in config): `"room_lights"`, `"wall_lights"`, etc.

This is the starting point for Tuya lights in the future module system.

4.2 Brain / Router (Future)

The brain/router will:

- Receive structured commands like:

    {
      "source": "telegram",
      "user_id": "<user>",
      "intent": "set_heat",
      "target": "primary_thermostat",
      "value": 22,
      "raw": "set heat to 22"
    }

- Look at:
  - `intent` (set_heat, plug_on, light_off, scene_run, etc.)
  - `target` (bed_lamp, room_lights, primary_thermostat, etc.)
- Consult the config files (`devices_*.json`) to see:
  - Which brand handles the target (tuya, nest, noma_iq, neewer).
  - Which device_id is behind that logical name.
- Call the appropriate module function:
  - `modules.nest.nest_thermostat.set_heat(DEVICE_ID, value)`
  - `modules.tuya.tuya_plugs.set_state(DEVICE_ID, True)`
  - `modules.tuya.tuya_lights.set_brightness(DEVICE_ID, 750)`
- Return a normalized response, for example:

    {
      "success": True,
      "intent": "set_heat",
      "target": "primary_thermostat",
      "module": "nest",
      "human_message": "Heat set to 22°C on primary thermostat.",
      "raw_details": {...}
    }

Interfaces (Telegram, CLI, etc.) then display `human_message` to the user.


4.3 Brand Modules (Future)

Each brand module will:

- Hide the ugly API details from the rest of the system.
- Provide a clean set of functions, for example:

**Tuya plugs (modules/tuya/tuya_plugs.py):**

- `set_plug_state(device_id: str, on: bool) -> dict`
- `get_plug_state(device_id: str) -> dict`

**Tuya lights (modules/tuya/tuya_lights.py):**

- `set_brightness(device_id: str, level: int)`
- `set_on(device_id: str)`
- `set_off(device_id: str)`

**Nest thermostat (modules/nest/nest_thermostat.py):**

- `set_heat(device_id: str, temp_c: float)`
- `get_temperature(device_id: str)`
- `get_hvac_status(device_id: str)`

These functions will reuse the logic that currently exists in things like
`ashpi_plug_test.py`, `ashpi_lights_tuya.py`, and `ashpi_nest_set.py`, but
organized into a consistent pattern.


4.4 Interfaces (Future)

We already have Telegram and CLI-like pieces – we just refine them.

- `interfaces/telegram_bot.py`
  - Parses text into structured commands.
  - Calls the brain/router.
  - Sends responses back to Telegram.

- `interfaces/cli.py`
  - Simple REPL for local testing:
    - `heat 22`
    - `plug bed_lamp on`
    - `lights room_lights 700`

- `interfaces/http_api.py` (future)
  - Offers a local HTTP endpoint for:
    - Mobile apps
    - Web UI
    - Other scripts


4.5 Config & Secrets (Future Organization)

- `config/devices_tuya.json`
  - Logical names → device IDs + metadata.

- `config/devices_nest.json`
  - Logical thermostat name → NEST_DEVICE path.

- `secrets/.env`
  - Holds:
    - Tuya Access ID / Secret / Region
    - Nest Client ID / Secret / Refresh Token
    - Telegram BOT_TOKEN
    - OpenAI API key
  - This file is NEVER committed anywhere public.

We will gradually migrate existing `env.py` and `ashpi_nest.env` into this
central place while keeping the old ones as historical references.


SECTION 5 – MIGRATION NOTES (FROM NOW TO FUTURE)
-------------------------------------------------

This section explains how the **current** scripts can be moved into the
future folder structure without losing your history.

5.1 Tuya Outlet

Current:
- `env.py` + `ashpi_plug_test.py`

Future:
- Move Tuya connection + commands into:
  - `modules/tuya/tuya_client.py` (connection)
  - `modules/tuya/tuya_plugs.py` (plug-specific logic)
- Replace direct CLI usage with calls from the brain/router.

5.2 Tuya Lights

Current:
- `ashpi_lights_tuya.py` + your devices snapshot + group ideas.

Future:
- Keep brightness and on/off logic and move it to:
  - `modules/tuya/tuya_lights.py`
- Use device/group definitions from:
  - `config/devices_tuya.json` instead of hardcoding.

5.3 Nest Thermostat

Current:
- `ashpi_nest.env`
- `ashpi_nest_set.py`
- Snapshot doc: `ashpi_nest_temperature_control_2025-12-08.txt`

Future:
- Move token fetch + executeCommand into:
  - `modules/nest/nest_client.py`
- Move `set_heat_celsius` into:
  - `modules/nest/nest_thermostat.py`
- Brain/router calls those methods when it receives a `set_heat` intent.

5.4 Telegram Bot

Current:
- Script that:
  - Parses text.
  - Directly calls Nest/Tuya functions.

Future:
- Convert it to:
  - Parse incoming text into a **command dict**.
  - Send dict to the brain/router.
  - Get normalized response back.
  - Send `human_message` to user.

This keeps Telegram simple and makes it brand-agnostic.
