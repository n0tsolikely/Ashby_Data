# **ASHBY PLATFORM ARCHITECTURE VISION**

### *Rules, Responsibilities, and Boundaries*

---

## 0. Purpose of This Document

This document exists to prevent Ashby from collapsing under its own ambition.

Ashby is a **platform**, not a single product.
This means:

* many modules
* many integrations
* many use-cases
* many future contributors (including future-you)

Without hard boundaries, everything becomes a god-system.

This document defines:

* **what belongs where**
* **what is forbidden**
* **how Ashby grows without refactors**
* **how Ashby stays shippable**

---

## 1. Core Principles (Non-Negotiable)

### 1.1 Engine vs Everything Else

* `engine/` is **sacred**
* Nothing vendor-specific lives in the engine
* Nothing product-specific lives in the engine
* Nothing optional lives in the engine

If a feature can be disabled, it **does not** belong in `engine/`.

---

### 1.2 WHAT vs HOW

* **WHAT** Ashby does в†’ `modules/`
* **HOW** it executes в†’ `adapters/`, `bridges/`, `native_vendor/`, `connectors/`

Violations of this rule are the fastest way to destroy the platform.

Modules:

* reason
* plan
* decide
* describe intent

Adapters / Providers:

* execute
* translate
* talk to APIs
* never reason

---

### 1.3 Scopes Are Universal

Scopes replace вЂњrooms,вЂќ вЂњzones,вЂќ вЂњprojects,вЂќ вЂњaccounts,вЂќ вЂњclients.вЂќ

Everything Ashby touches happens **inside a scope**.

* Physical spaces are *one* kind of scope
* Business domains are another
* Media pipelines are another

Nothing bypasses scopes.

---

### 1.4 Ashby Never Lies

Ashby only claims actions that were:

* executed
* verified
* confirmed

This is enforced by:

* `engine/truth/`
* tool verification
* state snapshots

---

## 2. Folder-by-Folder Rules

Below are **rules**, not suggestions.

---

## `engine/` вЂ” The Core Brain

### What belongs here

* routing
* cognition
* intent parsing
* policy
* memory
* truth enforcement
* eventing
* updates
* identity
* observability

### What is forbidden

* vendor APIs
* brand names
* device drivers
* hardcoded integrations
* feature logic

If you feel tempted to import something from `native_vendor/` or `bridges/` into `engine/`, **stop**.

---

## `engine/cognition/`

Pure intelligence.

* Intent в†’ meaning
* Meaning в†’ plan
* Plan в†’ tools

No execution happens here.

---

## `engine/eventing/`

This is the вЂњwhen X happens, do YвЂќ layer.

Used by:

* smart home automations
* business rules
* alerts
* future third-party logic

No device logic here. Only events.

---

## `engine/models/`

This is **AI managing AI**.

* model registry
* routing policy
* evaluation
* fallback

This is how Ashby stays future-proof when models change.

---

## `scopes/` вЂ” AshbyвЂ™s World Model

Scopes define:

* containment
* hierarchy
* permissions
* capabilities

AshbyвЂ™s zone logic **lives here**, not in modules.

---

## `domain/` вЂ” Abstract Reality

### Purpose

This layer describes *what exists* without caring *how itвЂ™s controlled*.

Examples:

* devices
* entities
* jobs
* assets
* capabilities

### Forbidden

* API calls
* OAuth
* SDKs

If you see HTTP here, something is wrong.

---

## `bridges/` вЂ” Temporary Integration Engines

This is where **Home Assistant**, **Matter**, **Zigbee** live.

### Rules

* Bridges are **replaceable**
* Bridges are **not depended on by engine**
* Bridges feed Ashby via providers

These folders are expected to shrink over time.

They are **strangler targets**.

---

## `native_vendor/` вЂ” Long-Term Ownership Path

This is where **AshbyвЂ™s own integrations** live.

Examples:

* tuya
* noma_iq_ayla
* nest_google_sdm
* reolink
* brand_4 / brand_5 / brand_6

### Rules

* Each vendor gets its own folder
* Each vendor implements the provider interface
* No cross-vendor logic

This is how Ashby eventually replaces bridges.

---

## `connectors/` вЂ” Non-Smart-Home Integrations

These are **business and productivity systems**, not devices.

Examples:

* email (gmail, outlook)
* chat (discord, slack)
* commerce (shopify, square)
* docs (drive, dropbox, notion)
* dev tools (github, jira)

### Rule

If itвЂ™s not a physical device, it probably belongs here.

---

## `modules/` вЂ” WHAT Ashby Does

Modules are **features**, not infrastructure.

Examples:

* lights
* comfort
* cameras
* image_gen
* stuart_meetings
* tori_inventory

### Rules

* Modules can be added/removed without breaking Ashby
* Modules never talk directly to vendors
* Modules register tools
* Modules own their own pipelines if needed

---

### Stuart (`modules/stuart_meetings/`)

Stuart is a **heavy module**:

* recording
* diarization
* transcription

---

## `adapters/` вЂ” Capability Execution

Adapters translate Ashby intent into actions.

Grouped by **capability**, not brand.

Examples:

* lights
* comfort
* cameras
* media

Adapters:

* never reason
* never store memory
* never make decisions
