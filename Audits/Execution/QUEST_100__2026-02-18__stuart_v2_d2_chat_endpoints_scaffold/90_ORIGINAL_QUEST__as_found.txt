    Create the stable chat endpoints without pretending we have LLM wired.

Goal:
- UI can send messages
- backend responds with a deterministic structured reply and optional plan preview

Later dungeons will replace the response generator with the real LLM + retrieval + citations.


    ------------------------------------------------
    SCOPE / OBJECTIVE
    ------------------------------------------------
    A) Implement POST /api/chat (session scope)

   Request (ChatRequestV1):
     {
       "session_id": "ses_...",
       "text": "string",
       "attachments": [ {filename,mime_type,size_bytes,sha256}? ]  (optional)
       "ui": { mode, template_id, retention, profile, ... }        (optional)
       "client": { "tab": "chat|session|outputs", "ts": ... }      (optional)
     }

   Response (ChatResponseV1) — MUST be stable:
     {
       "ok": true,
       "session_id": "...",
       "scope": "session",
       "reply": {
         "text": "string",
         "kind": "planner" | "system" | "not_implemented",
         "citations": [],          # empty for now (LLM later)
         "actions": [
           # optional: structured suggested actions, NOT auto-executed
           # e.g. {"type":"SUGGEST_RUN","endpoint":"/api/run","payload":{...}}
         ]
       },
       "planner": { ... } | null,  # include clarify_or_preview result when used
       "trace": { "request_id": "..." }
     }

   Implementation v1:
     - Use existing clarify_or_preview for commands like “formalize”, “run”, “help”.
     - Do NOT start runs inside /api/chat. It can only suggest /api/run payloads.

   Legacy:
     - Keep /api/message route as alias to /api/chat or keep old response but document deprecation.
     - Prefer adding /api/chat and updating UI client to use it in Dungeon 3.

B) Implement POST /api/chat/global

   v1 behavior:
     - Return ok=false NOT_IMPLEMENTED with correct envelope OR
     - implement trivial read-only help response (no retrieval).

   Must include:
     - scope="global"
     - clear disclosure that global chat is not implemented yet.

C) Tests
   - Test /api/chat returns ok=true and stable keys.
   - Test it does NOT create runs as side effect.
   - Test /api/chat/global returns NOT_IMPLEMENTED (or minimal ok=true system message).

D) Docs update
   - Contract doc includes chat request/response schema (even if not fully functional).
