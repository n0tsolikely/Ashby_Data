    ------------------------------------------------
    DESCRIPTION
    ------------------------------------------------
    Implement a standard response envelope + request/response models for the Stuart Web API.

This is how we avoid:
- a UI that breaks because endpoints disagree about field names
- silent “ok but actually failed” payloads
- impossible-to-debug error handling

This quest does NOT implement all endpoints — it creates the primitives
(models + helpers) that the other endpoint quests will use.

    ------------------------------------------------
    SCOPE / OBJECTIVE
    ------------------------------------------------
    A) Create shared envelope + error helpers

   Create a module (pick ONE canonical location; document it in Quest 090 doc):
     - ashby/interfaces/web/http_envelope.py   (recommended)
     OR
     - ashby/interfaces/web/api_common.py

   Must provide:
     - ApiError model:
         { code: str, message: str, details?: object }
     - success envelope helper:
         ok(payload: dict) -> dict  # ensures "ok": True
     - failure envelope helper:
         fail(code: str, message: str, status: int, details?: object) -> JSONResponse
         (or returns dict and caller wraps JSONResponse)
     - trace helpers (optional but recommended):
         - request_id generation (uuid4 short)
         - include in both success and error responses under "trace"

   Error code conventions (examples):
     - INVALID_REQUEST
     - NOT_FOUND
     - NOT_IMPLEMENTED
     - INTERNAL_ERROR

   HARD RULE:
     - Do NOT leak stack traces into responses by default.
     - If you include debug details, gate it behind env var (e.g., STUART_DEBUG=1).

B) Define Pydantic request/response models for core objects (v1)
   Create module:
     - ashby/interfaces/web/api_models_v1.py  (recommended)
     OR
     - ashby/interfaces/web/schemas_v1.py

   Models to define (minimum):
     - SessionSummaryV1
     - SessionDetailV1
     - RunSummaryV1
     - RunDetailV1
     - TranscriptVersionSummaryV1
     - TranscriptVersionV1 (full transcript payload)
     - FormalizationSummaryV1
     - ExportResponseV1
     - ChatRequestV1, ChatResponseV1 (scaffold; content can be minimal)
     - PaginationV1 (limit/offset/total? optional)

   NOTE:
     - Models must be compatible with our storage truth (runtime disk artifacts).
     - Do NOT invent fields that require cloud tenancy/auth.

C) Refactor existing endpoints to adopt envelope conventions (minimal touch)
   Minimum conversion targets:
     - /api/registry
     - /api/sessions (existing)
     - /api/upload
     - /api/run
     - /api/runs/{run_id}
     - /api/message (legacy)
   Do not rewrite logic; just normalize response shapes and errors.

D) Add tests that assert envelope consistency
   - New test file recommended:
     Ashby_Engine/tests/test_meetings_web_api_envelope_consistency.py
   - Ensure at least a few endpoints return {ok: True|False} and error envelope shape.
